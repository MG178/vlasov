# ПОЯСНИТЕЛЬНАЯ ЗАПИСКА

## к курсовому проекту по теме:
 «Проектирование базы данных и разработка клиентского приложения для автоматизации логистики курьерской доставки малогабаритных грузов»

---

# ВВЕДЕНИЕ

Стремительное развитие электронной коммерции и рост объёмов онлайн-торговли в последние годы сформировали устойчивый спрос на услуги курьерской доставки. По данным аналитических агентств, рынок экспресс-доставки в России ежегодно прирастает на 15–20%, при этом значительная доля участников рынка — небольшие региональные компании, не располагающие современными инструментами цифрового управления логистическими процессами.

В практике малых логистических компаний по-прежнему широко распространён ручной подход к организации доставки: заявки принимаются по телефону или в мессенджерах, маршруты курьеров планируются устно, а учёт выполненных заказов ведётся в таблицах Microsoft Excel. Такой подход неизбежно порождает системные проблемы: потеря или задвоение заказов, невозможность оперативного контроля местонахождения груза, отсутствие документального подтверждения факта вручения, высокая зависимость от «человеческого фактора».

Решением данных проблем является создание специализированной информационной системы, автоматизирующей весь жизненный цикл заказа — от его оформления клиентом до подтверждения получения грузополучателем с фиксацией геолокации и QR-идентификацией.

Настоящий курсовой проект посвящён проектированию и разработке информационной системы ExpressLog — веб-приложения для автоматизации курьерской доставки малогабаритных грузов (до 30 кг). В рамках проекта решаются следующие задачи:

- анализ предметной области и формализация бизнес-процессов курьерской доставки;
- разработка UML-диаграмм, описывающих требования к системе и взаимодействие её участников;
- проектирование нормализованной реляционной базы данных в третьей нормальной форме (3НФ) средствами MS SQL Server;
- реализация объектов базы данных: представлений, хранимых процедур, триггеров и пользовательских функций;
- разработка прототипа веб-приложения на основе архитектуры ASP.NET Core Web API с клиентской частью на React;
- разработка GitHub-репозитория с описанием проекта.

Практическая значимость проекта состоит в возможности его реального применения малыми и средними логистическими компаниями для перехода от ручных процессов к цифровому управлению доставкой, что позволяет сократить время обработки заказа, исключить потери и повысить прозрачность логистических операций.

# 1 АНАЛИЗ ТРЕБОВАНИЙ И ПОСТАНОВКА ЗАДАЧИ

## 1.1 Анализ предметной области

 1.1.1 Характеристика предметной области

Предметной областью данного проекта является процесс организации и управления курьерской доставкой малогабаритных грузов в условиях небольшой логистической компании. Под малогабаритными грузами понимаются отправления весом до 30 кг, допускающие транспортировку одним курьером без специального грузоподъёмного оборудования: документы, одежда, электроника, продукты питания и аналогичные категории товаров.

Основными участниками бизнес-процессов являются:

- Клиент (отправитель/получатель) — физическое или юридическое лицо, создающее заявку на доставку либо являющееся грузополучателем;
- Курьер — сотрудник компании, осуществляющий физическую транспортировку грузов в закреплённой зоне покрытия;
- Диспетчер/логист — оператор, принимающий и распределяющий заявки, контролирующий выполнение заказов в режиме реального времени;
- Администратор системы — специалист, обеспечивающий техническое администрирование платформы, управление пользователями и справочными данными.

 1.1.2 Анализ существующих процессов и их недостатки

На основании анализа технического задания был выделен типичный набор проблем, характерных для компаний, не использующих специализированные информационные системы:

Таблица 1.1 — Проблемы существующих процессов и способы их устранения

| № | Выявленная проблема | Последствие | Решение в системе |
|---|---|---|---|
| 1 | Приём заявок по телефону/WhatsApp | Потеря заказов, ошибки при записи | Форма создания заказа с валидацией |
| 2 | Учёт в таблицах Excel | Дублирование данных, нет актуального статуса | Централизованная БД с ролевым доступом |
| 3 | Голосовые отчёты курьеров | Нет документального подтверждения доставки | QR-подтверждение с геолокацией |
| 4 | Ручной расчёт стоимости | Ошибки тарификации | Автоматический расчёт через функцию БД |
| 5 | Нет трекинга груза | Клиент не знает, где посылка | Смена статусов + история геолокаций |
| 6 | Нет аналитики по курьерам | Невозможно оценить эффективность | VIEW агрегации показателей |

 1.1.3 Цель и задачи разрабатываемой системы

Цель проекта — создание информационной системы ExpressLog, автоматизирующей полный цикл курьерской доставки малогабаритных грузов.

Для достижения указанной цели необходимо решить следующие задачи:

1. Автоматизировать приём, регистрацию и распределение заказов между курьерами с учётом зон покрытия и загруженности.
2. Обеспечить прозрачность перемещения груза посредством смены статусов и фиксации геолокации.
3. Реализовать механизм подтверждения факта вручения груза через QR-код с контролем геозоны.
4. Предоставить личные кабинеты всем категориям пользователей с разграниченным доступом.
5. Сформировать инструменты отчётности и аналитики для диспетчеров и руководства.

 1.1.4 Границы системы и ограничения

В рамках проекта реализуется автоматизация доставки только по территории, разделённой на три зоны (Центр, Пригород, Область). Система поддерживает работу с грузами весом до 30 кг. Интеграция с картографическими сервисами реального времени (Google Maps, 2GIS) носит имитационный характер и реализована через фиксацию координат в базе данных. Оплата заказов в системе не обрабатывается — фиксируется только итоговая стоимость для бухгалтерского учёта.

## 1.2 Разработка UML-диаграмм

 1.2.1 Диаграмма вариантов использования (Use Case)

Диаграмма вариантов использования описывает функциональные требования к системе с точки зрения её пользователей — акторов. Для системы ExpressLog выделены четыре основных актора и набор вариантов использования, распределённых между ними.

Таблица 1.2 — Акторы системы и их краткое описание

| Актор | Роль в системе | Основные функции |
|---|---|---|
| Клиент | Отправитель или получатель груза | Создание заказа, трекинг, подтверждение получения |
| Курьер | Исполнитель доставки | Просмотр маршрута, смена статусов, QR-отображение |
| Диспетчер | Оператор управления | Назначение курьеров, контроль заказов, отчёты |
| Администратор | Системный администратор | Управление пользователями, справочниками, настройки |

Система включает следующие основные варианты использования:

- «Зарегистрироваться / Авторизоваться» — общий для всех акторов;
- «Создать заказ» — доступен Клиенту и Диспетчеру;
- «Рассчитать стоимость» — включается в «Создать заказ» (связь `<<include>>`);
- «Отследить заказ» — доступен Клиенту, Диспетчеру;
- «Назначить курьера» — доступен Диспетчеру, расширяется автоматическим назначением (связь `<<extend>>`);
- «Просмотреть маршрутный лист» — доступен Курьеру;
- «Изменить статус заказа» — доступен Курьеру;
- «Подтвердить доставку (QR)» — доступен Клиенту и Курьеру;
- «Сформировать отчёт» — доступен Диспетчеру и Администратору;
- «Управлять пользователями» — доступен только Администратору;
- «Настроить тарифы» — доступен только Администратору.

 1.2.2 Диаграмма последовательности (Sequence Diagram)

Диаграмма последовательности описывает порядок взаимодействия объектов системы во времени для конкретного сценария. Рассмотрен ключевой бизнес-сценарий системы: «Создание заказа, назначение курьера и подтверждение доставки через QR-код».

Таблица 1.3 — Роли и участники диаграммы последовательности

| Участник (объект) | Тип | Описание роли в сценарии |
|---|---|---|
| Клиент | Актор | Инициирует создание заказа, подтверждает получение |
| Веб-интерфейс | Граница (Boundary) | Отображает формы, принимает ввод пользователя |
| API-контроллер | Управление (Control) | Обрабатывает HTTP-запросы, вызывает сервисы |
| OrderService | Сервис | Содержит бизнес-логику работы с заказами |
| База данных | Сущность (Entity) | Хранит и возвращает данные |
| QRService | Сервис | Генерирует токен и QR-изображение |
| Курьер | Актор | Выполняет доставку, отображает QR получателю |

Сценарий взаимодействия разворачивается в следующей последовательности шагов:

1. Клиент заполняет форму создания заказа в веб-интерфейсе, указывая адреса, тип груза, вес и срочность.
2. Веб-интерфейс отправляет POST-запрос на эндпоинт `/api/orders` контроллера.
3. API-контроллер передаёт данные в `OrderService`, который вызывает функцию `fn_РассчитатьСтоимость` в базе данных.
4. База данных возвращает рассчитанную стоимость; сервис создаёт запись заказа со статусом «Новый» и присваивает уникальный трек-номер.
5. Диспетчер (либо автоматически — хранимая процедура `usp_НазначитьКурьера`) назначает курьера. Процедура находит свободного курьера в нужной зоне, генерирует QR-токен и создаёт сессию доставки.
6. Курьер получает уведомление и просматривает маршрутный лист. После получения груза он меняет статус на «Взят_курьером» и статус меняется на «В_пути».
7. По прибытии на адрес получателя курьер отображает QR-код в мобильном интерфейсе.
8. Клиент сканирует QR-код камерой телефона. Фронтенд направляет токен на сервер вместе с геолокацией устройства.
9. Сервер валидирует токен, сравнивает координаты с адресом доставки. При успехе создаётся запись в `DeliveryConfirmations`, триггер разрешает вставку в `OrderStatusHistory` со статусом «Доставлен».
10. Клиент и диспетчер получают push-уведомления об успешной доставке.

 1.2.3 Диаграмма классов (Class Diagram)

Диаграмма классов отражает структуру объектной модели системы — сущности предметной области, их атрибуты и взаимосвязи. В контексте проекта диаграмма классов используется как основа для построения физической модели базы данных и разработки C#-сущностей Entity Framework Core.

Основные классы модели:

- `User` — базовая сущность пользователя, от которой наследуются `Client` и `Courier`;
- `Order` — центральная сущность, связанная с большинством других классов;
- `DeliverySession` — хранит QR-токен и управляет жизненным циклом подтверждения;
- `DeliveryConfirmation` — фиксирует факт вручения груза;
- `OrderStatusHistory` — ведёт аудит всех изменений статуса заказа.

Связи между классами:

- `User` — `Order` (1:N): один клиент может создать множество заказов;
- `Courier` — `Order` (1:N): один курьер может быть назначен на несколько заказов;
- `Order` — `DeliverySession` (1:1): каждый заказ имеет не более одной активной сессии;
- `DeliverySession` — `DeliveryConfirmation` (1:1): сессия завершается одним успешным подтверждением;
- `Order` — `OrderStatusHistory` (1:N): заказ имеет историю из нескольких записей статусов.

## 1.3 Разработка макета приложения

Макет приложения (прототип пользовательского интерфейса) разрабатывается с учётом ролевой модели системы. Каждый тип пользователя получает отдельный личный кабинет с набором экранов, соответствующих его функциональным обязанностям.

 1.3.1 Экран авторизации

Стартовый экран системы представляет форму входа, содержащую поля «Email» и «Пароль», кнопку «Войти», а также ссылку «Забыли пароль?». После успешной аутентификации система определяет роль пользователя по JWT-токену и перенаправляет его на соответствующий дашборд. Для незарегистрированных пользователей предусмотрена кнопка «Отследить заказ по трек-номеру», позволяющая проверить статус без авторизации.

 1.3.2 Личный кабинет клиента

Кабинет клиента включает следующие разделы: «Новый заказ», «Мои заказы», «Отследить». Главный экран — форма создания заказа с встроенным калькулятором стоимости. При получении груза клиент нажимает «Подтвердить получение» и сканирует QR-код курьера камерой телефона.

## 2.1 Разработка ER-диаграммы на основе сырых данных

 2.1.1 Анализ исходных данных

На этапе начального проектирования базы данных были проанализированы исходные данные, предоставленные в техническом задании: справочники типов грузов и тарифов в формате JSON, зоны доставки и курьеры в формате CSV, а также требования к начальным записям заказов и QR-сессий.

Из предоставленных данных были выделены следующие сущности:

Таблица 2.1 — Выявленные сущности из сырых данных

| Источник данных | Выявленная сущность | Ключевые атрибуты |
|---|---|---|
| JSON типов грузов | CargoTypes | type_id, name, max_weight, fragile |
| CSV зон | DeliveryZones | id, name, city, coefficient |
| CSV курьеров | Couriers | id, name, transport, zone_id, rating |
| Описание заказов | Orders | track_number, addresses, status, cost |
| QR-сессии | DeliverySessions | session_id, token, order_id, status |

 2.1.2 Логическая ER-диаграмма

Логическая модель описывает сущности, их атрибуты и взаимосвязи без привязки к конкретной СУБД. Основу логической модели составляет сущность Orders (Заказы), которая связана отношениями «многие-к-одному» с сущностями Clients (клиент-отправитель), Couriers (исполнитель), CargoTypes (тип груза) и DeliveryZones (зона доставки). Сущность OrderStatusHistory связана с Orders отношением «один-ко-многим», фиксируя историю всех изменений статуса. Сущность DeliverySessions связана с Orders отношением «один-к-одному» и порождает единственную запись в DeliveryConfirmations при успешном подтверждении.

## 2.2 Разработка физической модели базы данных в MS SQL Server

 2.2.1 Описание таблиц базы данных

В данном разделе приведены SQL-скрипты создания всех таблиц базы данных, реализующие физическую модель, рассмотренную в п. 2.1.3.

Таблица Users — Пользователи системы

Общая таблица для всех категорий пользователей. Реализует ролевую модель через атрибут `Role`.

```sql
CREATE TABLE Users (
    UserID        INT           IDENTITY(1,1) PRIMARY KEY,
    FullName      NVARCHAR(150) NOT NULL,
    Email         NVARCHAR(100) NOT NULL,
    Phone         NVARCHAR(20)  NOT NULL,
    PasswordHash  NVARCHAR(256) NOT NULL,
    Role          NVARCHAR(20)  NOT NULL
                  CONSTRAINT CHK_Users_Role
                  CHECK (Role IN ('Client','Courier','Dispatcher','Admin')),
    IsActive      BIT           NOT NULL DEFAULT 1,
    RegisteredAt  DATETIME      NOT NULL DEFAULT GETDATE(),
    CONSTRAINT UQ_Users_Email UNIQUE (Email),
    CONSTRAINT UQ_Users_Phone UNIQUE (Phone)
);
```

Таблица Clients — Расширение профиля клиента

```sql
CREATE TABLE Clients (
```

Таблица Tariffs — Тарифы на перевозку

```sql
CREATE TABLE Tariffs (
    TariffID        INT           IDENTITY(1,1) PRIMARY KEY,
    CargoTypeID     INT           NOT NULL,
    ZoneID          INT           NOT NULL,
    PricePerKg      DECIMAL(10,2) NOT NULL,
    PricePerKm      DECIMAL(10,2) NOT NULL,
    UrgentSurcharge DECIMAL(10,2) NOT NULL DEFAULT 0,
```

Таблица предназначена для хранения оценок и комментариев клиентов по завершённым заказам. Оценка задаётся по шкале от 1 до 5 с CHECK-ограничением. На основе значений этой таблицы рассчитывается рейтинг курьера в представлении `vw_ЭффективностьКурьеров`.

Таблица Notifications — Уведомления

### 2.3.5 Программные объекты базы данных

Представление vw_ЭффективностьКурьеров

Представление агрегирует данные из таблиц `Orders`, `OrderStatusHistory`, `DeliveryConfirmations` и `ClientReviews` для формирования сводной статистики по каждому курьеру. Вычисляются: количество заказов за текущий день, среднее время доставки (разница между временем назначения и временем подтверждения), процент просроченных заказов и средняя оценка по отзывам. Представление используется на странице отчётности диспетчера и при расчёте рейтинга курьера.

```sql
CREATE VIEW vw_ЭффективностьКурьеров AS
SELECT
    c.CourierID,
    u.FullName                                        AS КурьерФИО,
    COUNT(o.OrderID)                                  AS ВсегоЗаказовСегодня,
    AVG(DATEDIFF(MINUTE,
        o.CreatedDate,
        dc.ConfirmationTime))                         AS СреднееВремяДоставкиМин,
    CAST(
        SUM(CASE WHEN o.DesiredDeliveryDate IS NOT NULL
                  AND dc.ConfirmationTime > o.DesiredDeliveryDate
             THEN 1 ELSE 0 END)  100.0
        / NULLIF(COUNT(o.OrderID), 0)
    AS DECIMAL(5,2))                                  AS ПроцентПросрочек,
    AVG(CAST(cr.Rating AS DECIMAL(3,2)))              AS СредняяОценка
FROM Couriers c
JOIN Users u         ON c.CourierID = u.UserID
LEFT JOIN Orders o   ON o.AssignedCourierID = c.CourierID
                    AND CAST(o.CreatedDate AS DATE) = CAST(GETDATE() AS DATE)
LEFT JOIN DeliveryConfirmations dc ON dc.OrderID = o.OrderID
LEFT JOIN ClientReviews cr         ON cr.OrderID = o.OrderID
GROUP BY c.CourierID, u.FullName;
```

Хранимая процедура usp_НазначитьКурьера

Процедура принимает идентификатор заказа в качестве входного параметра. Алгоритм работы: определяется зона доставки заказа; среди курьеров этой зоны выбирается тот, у кого активных заказов (статус не «Доставлен» и не «Отменен») менее пяти и кто работает сегодня. Если свободный курьер найден — обновляется статус заказа на «Назначен», генерируется уникальный GUID-токен и создаётся запись в `DeliverySessions`. В случае отсутствия свободных курьеров процедура возвращает информационное сообщение об ошибке.

Триггер trgr_ПроверкаДоставлен (INSTEAD OF INSERT)

Триггер установлен на таблицу `OrderStatusHistory`. Перед каждой попыткой добавить запись со статусом «Доставлен» выполняются две проверки:

1. Для данного заказа должна существовать запись в `DeliveryConfirmations` (факт сканирования QR);
2. Геолокация из записи подтверждения должна совпадать с координатами адреса доставки в пределах заданного радиуса (реализовано через вычисление приближённого расстояния по формуле разности координат; для учебного проекта допустимо отклонение в 0,001 градуса, что соответствует примерно 100 метрам).

Если обе проверки пройдены — строка вставляется в таблицу обычным образом (`INSERT`). Если хотя бы одна проверка не пройдена — триггер генерирует ошибку через `RAISERROR` и откатывает транзакцию.

Функция fn_РассчитатьСтоимость

Скалярная пользовательская функция принимает четыре параметра: `@CargoTypeID`, `@Weight`, `@ZoneID`, `@IsUrgent`. Из таблицы `Tariffs` извлекается соответствующий тариф. Итоговая стоимость рассчитывается по формуле:

$$\text{Стоимость} = (\text{PricePerKg} \times \text{Weight} + \text{PricePerKm} \times \text{DistanceKm}) \times \text{PriceCoefficient} + \text{UrgentSurcharge}$$

где `DistanceKm` — имитационное расстояние, задаваемое как фиксированное значение для каждой зоны (Центр — 5 км, Пригород — 15 км, Область — 40 км) в отсутствие реального картографического API. Функция вызывается при создании заказа через сервисный слой бэкенда и при работе калькулятора стоимости на фронтенде.

### 2.3.6 Тестовые данные

Для отладки и демонстрации работы системы в базу данных вносится набор начальных тестовых данных. В справочник зон доставки добавляются три записи: «Центр» (коэффициент 1,0), «Пригород» (1,5) и «Область» (2,0). В справочник типов грузов — четыре записи согласно техническому заданию: «Документы», «Одежда/текстиль», «Техника/электроника», «Продукты». Создаются три курьера (Петров А.А., Сидоров М.И., Иванов С.П.) с различным транспортом и зонами покрытия. Формируется 12 тестовых заказов за последние 7 дней с различными статусами — четыре в статусе «Новый», три в «В_пути», три в «Доставлен», два в «Отменен». Для заказов в статусе «Доставлен» заполняются соответствующие записи в `OrderStatusHistory`, `DeliverySessions` и `DeliveryConfirmations`. Для текущего дня создаётся 5 активных QR-сессий.

# 3 ПРОЕКТИРОВАНИЕ И РАЗРАБОТКА ПРИЛОЖЕНИЯ

## 3.1 Разработка прототипа приложения

### 3.1.1 Архитектура приложения

Приложение ExpressLog построено по многоуровневой (слоистой) архитектуре, которая обеспечивает чёткое разделение ответственности между компонентами системы и упрощает её сопровождение и масштабирование.

[рисунок]

Рисунок 3.1 — Архитектурная схема приложения ExpressLog

Архитектура включает следующие уровни:

- Уровень представления (Frontend) — одностраничное веб-приложение на библиотеке React, взаимодействующее с сервером исключительно через REST API;
- Уровень API (Controllers) — набор контроллеров ASP.NET Core Web API, принимающих HTTP-запросы, выполняющих авторизацию по JWT-токену и делегирующих логику сервисному уровню;
- Уровень бизнес-логики (Services) — C#-классы, содержащие бизнес-правила: расчёт стоимости, валидацию геолокации, генерацию QR-кода, логику назначения курьера;
- Уровень доступа к данным (Repositories + DbContext) — взаимодействие с базой данных через Entity Framework Core; хранимые процедуры и функции вызываются через `FromSqlRaw` / `ExecuteSqlRaw`;
- База данных — MS SQL Server, содержащая всю деловую информацию и программные объекты, описанные в главе 2.

### 3.1.3 Реализация ключевых функций фронтенда

Модуль авторизации

При старте приложения React проверяет наличие JWT-токена в `localStorage`. Если токен действителен — пользователь перенаправляется на соответствующий дашборд. Недействительный или отсутствующий токен инициирует переход на экран авторизации. Все защищённые маршруты обёрнуты в компонент `PrivateRoute`, который выполняет проверку роли перед рендерингом страницы.

Калькулятор стоимости

Форма создания заказа содержит компонент `CostCalculator`. При изменении полей «Тип груза», «Вес», «Зона» или «Срочность» компонент отправляет GET-запрос на эндпоинт `/api/tariffs/calculate` с соответствующими параметрами и отображает возвращённую стоимость в режиме реального времени без перезагрузки страницы.

Модуль QR-сканирования

Для сканирования QR-кода используется библиотека `Html5Qrcode`. При нажатии кнопки «Подтвердить получение» запрашивается разрешение на использование камеры устройства. После успешного распознавания QR-кода извлечённый токен вместе с текущими координатами браузера (Geolocation API) отправляется POST-запросом на `/api/delivery/confirm`. В случае успеха интерфейс обновляет статус заказа на «Доставлен» и отображает уведомление.

Kanban-доска диспетчера

Доска реализована с использованием библиотеки `@dnd-kit`. Каждая колонка соответствует одному из статусов заказа. Карточки заказов содержат трек-номер, имя получателя, адрес и имя назначенного курьера. При перетаскивании карточки в другую колонку отправляется PATCH-запрос на изменение статуса. Кнопка «Назначить курьера» открывает модальное окно со списком свободных курьеров в нужной зоне.

# ЗАКЛЮЧЕНИЕ

В ходе выполнения курсового проекта была спроектирована и частично реализована информационная система ExpressLog, предназначенная для автоматизации процессов курьерской доставки малогабаритных грузов. Работа охватила полный проектный цикл — от анализа предметной области до разработки прототипа приложения и его документирования в репозитории GitHub.

В рамках первой главы был проведён анализ предметной области, в результате которого были формализованы основные бизнес-процессы логистической компании и выявлены системные проблемы ручного подхода к управлению доставкой. Разработан комплект UML-диаграмм, включающий диаграмму вариантов использования, диаграмму последовательности для ключевого сценария QR-подтверждения доставки и диаграмму классов объектной модели. Создан макет пользовательского интерфейса для всех четырёх ролей: клиента, курьера, диспетчера и администратора.

В рамках второй главы спроектирована нормализованная реляционная база данных в третьей нормальной форме, содержащая 15 таблиц. Для каждой таблицы определены типы данных атрибутов, первичные и внешние ключи, CHECK- и UNIQUE-ограничения. Разработаны и реализованы четыре программных объекта базы данных: представление `vw_ЭффективностьКурьеров` для аналитики, хранимая процедура `usp_НазначитьКурьера` для автоматического распределения заказов, триггер `trgr_ПроверкаДоставлен` для контроля целостности при смене статуса и скалярная функция `fn_РассчитатьСтоимость` для автоматической тарификации. Сформирован набор тестовых данных, достаточный для демонстрации работоспособности всех реализованных механизмов.

В рамках третьей главы описана слоистая архитектура веб-приложения на стеке ASP.NET Core Web API + React, определён полный перечень API-эндпоинтов с указанием методов, URL и ролей доступа. Описана реализация ключевых функций фронтенда: калькулятора стоимости, Kanban-доски диспетчера и модуля QR-сканирования. Структурирован GitHub-репозиторий с файлом README, содержащим инструкцию по развёртыванию проекта.

Таким образом, все задачи, поставленные во введении, выполнены в полном объёме. Разработанная система решает ключевые проблемы ручного управления доставкой: исключается потеря заказов за счёт централизованного хранения, обеспечивается прозрачность перемещения груза через смену статусов с геолокацией, а факт вручения фиксируется документально посредством QR-идентификации с проверкой геозоны.

Практическая значимость проекта состоит в его применимости в условиях реально действующей малой логистической компании. Дальнейшее развитие системы может включать интеграцию с картографическим API для отображения маршрутов в реальном времени, реализацию мобильного приложения для курьеров на React Native, внедрение push-уведомлений через Firebase Cloud Messaging, а также подключение к платёжному шлюзу для онлайн-оплаты заказов.
